MAS Sample Evolution: UX Storyboard

Four directions for making a recursive MAS run understandable. The recurring thesis: the user wants to debug how the sample evolved, so the graph should reveal one useful moment at a time instead of trying to expose every nested task at once.

timeline-first focused expansion details in inspector never overlap
Design A · Recommended default

Visual Debugger: current graph + timeline scrubber

Best when the primary question is “what happened next?” The graph shows the selected moment; the bottom rail controls time and explains the causal event.

Design C · Summary-first

Phase map: group noisy mutations into readable chapters

Best for post-run analysis and screenshots. Instead of exposing every mutation, it groups the run into phases and lets the user open a phase to inspect details.

Design D · Event-centric

Event filmstrip: timeline list drives a small graph viewport

Best when debugging is very event-driven. The left filmstrip is the source of truth; the graph is a small preview of the selected event’s affected nodes.
