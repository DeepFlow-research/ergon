# Async RL Architecture Discussion

This document captures a high-level architecture discussion for evolving Arcane from a benchmark/evaluation runtime into an asynchronous RL and post-training framework.

This is intentionally a discussion document, not an implementation spec.

The goal is to create a shared design reference for a large future architecture decision.

Companion documents:

- `design/async_rl/async_rl_library_implementation_review.md`: implementation-grounded review of cloned async RL libraries, the patterns they actually use, and the concrete Arcane design decisions that fall out of that review.

## Why This Matters

Today Arcane is primarily oriented around:

- benchmark execution
- sandboxed tool use
- evaluation and scoring
- dashboard/debug visibility

The longer-term goal is much bigger:

- run local or self-hosted models as rollout policies
- capture exact observations and generations losslessly
- compute rewards from benchmark environments and evaluators
- use those trajectories to post-train local models
- support asynchronous policy updates and ongoing rollout generation

If that is the goal, then the architecture should be designed now so that:

- benchmarks remain modern and decoupled from core runtime logic
- frontends remain useful for debugging and experiment inspection
- the runtime can evolve into a distributed actor-learner system
- Arcane does not become tightly coupled to one specific RL library or serving stack

## Core Thesis

Arcane should not become "an RL algorithm implementation."

Arcane should become:

- the environment and orchestration layer
- the canonical trace and reward store
- the experiment control plane
- the benchmark and evaluator runtime
- the debug and inspection frontend

And then it should integrate with:

- a generator service for inference
- a trainer backend for post-training

This lets Arcane own the hard systems problem without forcing it to reimplement PPO, GRPO, or other post-training algorithms from scratch.

## The Right Mental Model

The system naturally separates into three major planes.

### 1. Environment plane

This is Arcane itself.

Responsibilities:

- benchmark tasks
- tool execution
- sandbox execution
- stakeholder simulators
- evaluators
- reward computation
- state persistence
- trace capture

This is where observations and rewards are defined.

### 2. Serving plane

This is the inference system for the current policy.

Examples:

- `vLLM`
- another local inference server
- a self-hosted generation service

Responsibilities:

- serve a specific policy version
- generate actions from observations
- optionally expose logprobs and streaming deltas

### 3. Learning plane

This is the post-training system.

Examples:

- TRL
- OpenRLHF
- veRL
- a custom trainer

Responsibilities:

- consume trajectories
- compute losses and updates
- produce new checkpoints
- register new policy versions

## MDP Framing

The most useful first MDP boundary is probably not token-level.

The first practical abstraction should be turn-level.

### Observation

Everything visible to the policy before one model call, including:

- user/task prompt
- history
- tool results
- environment outputs
- relevant resources

### Action

One assistant/model turn generated by the policy.

This may include:

- plain text
- tool calls
- structured outputs

### Transition

The environment consumes the model turn and produces:

- tool side effects
- new observations
- updated task state

### Reward

A reward signal computed from:

- benchmark evaluation
- rule-based shaping
- outcome success/failure
- optional human feedback later

This means the first useful training abstraction is likely turn-level RL with token metadata attached, not a pure token-as-action environment.

## What Arcane Should Store

If the long-term goal is asynchronous RL and local post-training, then the current `actions` table is the wrong source of truth.

The canonical store should capture:

- exact model request content
- exact model response content
- exact streamed deltas where available
- exact tool call arguments
- exact tool return payloads
- exact observations that become visible to the policy
- exact reward and terminal outcome signals
- exact ordering and timing
- exact provider/model/settings metadata

This is why a `trace_events` model is a better fit than `actions`.

## Recommended Storage Strategy

Use different systems for different data classes.

### Postgres

Use Postgres for:

- run metadata
- rollout episode metadata
- rollout group metadata
- policy version metadata
- canonical trace event index
- reward events
- trainer job metadata
- checkpoint metadata and artifact pointers

Postgres should store the authoritative relational view of the system.

### Blob / object storage / local filesystem artifacts

Use artifact storage for:

- large trace payloads if needed
- binary resources
- checkpoints
- optimizer states
- potentially detailed token-level caches

Do not store large model states directly in Postgres.

### Redis

Use Redis for:

- queues
- leases
- inflight coordination
- active policy pointer caches
- transient scheduling state

Redis should not be the canonical long-term trajectory store.

## Policy Versioning

Asynchronous RL only works cleanly if every rollout is tied to a specific immutable policy version.

That means the system needs a first-class notion of:

- `policy_version`

Every rollout / episode / trace should record:

- which policy version produced it
- which benchmark version/environment version it used
- which serving config was active

That makes later training and analysis much cleaner.

## Avoid Global Freezes

One tempting approach is:

- wait for all workflows to finish
- stop serving
- train
- reload the model
- resume serving

This is not the right default architecture.

The better architecture is versioned asynchronous rollouts:

1. policy version `v_k` serves rollouts
2. completed trajectories from `v_k` accumulate
3. the trainer consumes a selected training window from `v_k`
4. the trainer produces `v_{k+1}`
5. serving switches new rollouts to `v_{k+1}`
6. inflight `v_k` rollouts are allowed to complete

This avoids freezing the entire system and matches modern actor-learner design much better.

## Recommended Core Components

### 1. Rollout Scheduler

Responsibilities:

- select tasks/prompts to run
- create rollout jobs
- attach benchmark/environment config
- attach policy version
- support grouped sampling for methods like GRPO

### 2. Rollout Worker

Responsibilities:

- pull a rollout job
- interact with Arcane environment state
- request generation from the policy server
- execute tools and environment transitions
- emit trace events
- finalize rewards and episode status

This is the place where existing worker abstractions can evolve into a rollout engine.

### 3. Generator Service

Responsibilities:

- serve a model checkpoint
- generate completions
- optionally expose logprobs
- optionally expose streaming deltas

Examples:

- `vLLM`
- another self-hosted model server

Arcane should talk to this behind a generator interface.

### 4. Environment Runtime

This remains Arcane’s core responsibility.

Responsibilities:

- benchmarks
- tools
- sandboxes
- evaluators
- stakeholders
- reward logic
- trace capture

### 5. Trace Store

Responsibilities:

- canonical append-only event log
- exact runtime record
- later replay, export, and post-training extraction

### 6. Learner Orchestrator

Responsibilities:

- decide when enough trajectories are ready
- assemble training windows
- call trainer backends
- register new policy versions
- trigger serving updates

### 7. Trainer Adapter

Responsibilities:

- bridge Arcane trajectory data to an external training framework
- invoke training
- return policy artifacts and metrics

This is the key interface for supporting bring-your-own-framework training.

### 8. Policy Registry

Responsibilities:

- store version metadata
- map aliases like "active" or "candidate" to real versions
- track lineage between policy versions

### 9. Serving Controller

Responsibilities:

- deploy or swap the active policy version in the generator service
- health-check serving
- perform blue/green or rolling swaps

## Interfaces We Should Draw Clearly

These are the boundaries where abstraction matters most.

### `Generator`

Arcane should not care whether generation comes from:

- cloud APIs
- `vLLM`
- local inference
- another backend

The generator interface should abstract that away.

Suggested responsibilities:

- generate from an observation
- optionally return logprobs
- identify policy version used

### `Environment`

Benchmarks should continue moving toward a common environment interface.

Even if not modeled exactly like Gym, a useful abstraction is:

- reset
- step
- terminal check
- reward extraction
- observation serialization

### `TrainerAdapter`

Arcane should not implement GRPO, PPO, etc. directly if it can avoid it.

Instead, it should define a trainer adapter boundary that allows:

- TRL
- OpenRLHF
- veRL
- custom research code

to consume Arcane-generated trajectories.

### `TraceExporter`

There should eventually be a separate trajectory export layer that can derive:

- turn-level trajectories
- token-level sequences
- grouped GRPO samples
- evaluation/reward windows

without coupling that logic to the online runtime.

## Why This Helps With GRPO And Similar Methods

If the first target is an advantage-style method like GRPO, the natural unit of scheduling is often not just a single episode.

It is often:

- a rollout group for the same prompt/task
- with multiple sampled completions from the same frozen policy version

That implies additional entities like:

- `rollout_group`
- `rollout_episode`

Each should record:

- prompt/task identity
- policy version
- benchmark/environment version
- reward outcomes

This should be supported at the scheduler/data-model level rather than hacked in later.

## Deployment Topology

### Local / developer setup

For local iteration, `docker compose` is likely enough.

It can provision:

- Postgres
- Redis
- Arcane backend
- Arcane dashboard
- one `vLLM` instance
- optionally one trainer container

This is good for:

- architecture validation
- debugging
- small-scale experiments

### Real distributed setup

For serious rollout/training throughput, Compose should not be the long-term orchestration layer.

You will likely want:

- Ray
- Kubernetes
- Slurm

depending on whether the target is production-style infra or research/HPC infra.

### Recommended stance

- use containers for everything
- use Compose for local development
- design services so they can later be scheduled by a real distributed control plane

## Multi-GPU And Multi-Node Considerations

There are at least three realistic topologies.

### A. Separate rollout GPUs and training GPUs

Best long-term design.

- one resource pool serves the active model
- another resource pool trains new checkpoints
- Arcane orchestrates between them

### B. Shared GPUs with sequential reuse

Useful early on.

- serve
- pause
- train
- redeploy

This is viable for small setups but should not define the core architecture.

### C. Large-model serving with tensor parallelism

If the rollout model is large, the generator service may need:

- tensor parallelism
- pipeline parallelism
- multi-node serving

That is another reason to keep serving behind a networked generator abstraction rather than baking local model execution into Arcane itself.

## Data Volume / Dimensional Analysis

One important question is whether this design implies unmanageable storage.

The answer is: it can, if done naively.

### What makes storage explode

- storing the full growing prompt history inline on every step
- storing every binary artifact inline in Postgres
- storing top-k token logprobs for every token forever
- storing optimizer state in the relational database

### What is probably manageable

- event-ordered text payloads
- structured tool payloads
- references to binary artifacts
- policy metadata and rollout metadata in Postgres
- checkpoints and optimizer state outside Postgres

### Practical implication

We should design for:

- canonical trace events in Postgres
- large blobs in artifact/object storage
- optional retention tiers
- optional downsampled or projected UI views

## Inspirations Worth Pulling From

There are a few system patterns worth studying more deeply.

### SEED RL / IMPALA-style actor-learner systems

Useful ideas:

- actor-learner separation
- asynchronous rollouts
- versioned policies
- no full-system freeze

### OpenRLHF / vLLM-style serving + training separation

Useful ideas:

- serving and training as separate systems
- checkpoint/version handoff
- inference as the throughput bottleneck

### Agent Lightning / trace-based training systems

Useful ideas:

- span/trace-based capture
- decoupling runtime instrumentation from training logic
- building RL on top of traces rather than rewriting agent logic

## Strong Recommendations

### Recommendation 1

Treat Arcane as:

- environment runtime
- trace store
- evaluator/reward engine
- control plane
- frontend/debug layer

not as the place to implement every RL algorithm directly.

### Recommendation 2

Make `trace_events` the source of truth and keep the runtime lossless enough for:

- replay
- export
- later post-training

### Recommendation 3

Add policy versioning now, even before online RL is implemented fully.

### Recommendation 4

Design a `TrainerAdapter` interface early so users can bring:

- TRL
- OpenRLHF
- veRL
- custom trainers

without Arcane needing to implement GRPO from scratch.

### Recommendation 5

Keep Compose as a dev topology only. Design the services so they can later run under a real distributed scheduler.

## Open Questions

These are the major unresolved questions that deserve follow-up design work.

### 1. What is the exact observation boundary?

Do we train on:

- full turn observations
- tool-call substeps
- token-level deltas

### 2. How much token metadata do we persist?

Do we store:

- only raw text and events
- chosen-token logprobs
- top-k distributions

### 3. Where should trace payload thresholds live?

At what point do we spill large payloads out of Postgres into artifact storage?

### 4. What is the first trainer backend we want to support?

The first adapter target strongly shapes the initial export format.

### 5. What is the first rollout topology we want to support?

Do we optimize first for:

- local single-node dev
- one serving node + one training node
- multi-node distributed serving

### 6. How should the dashboard present raw trace vs grouped trace?

A lossless event log is excellent for training and debugging, but probably too verbose for a naive UI.

## Suggested Next Steps

1. Write a dedicated proposal for the `trace_events` schema and associated relational entities such as:
   - `policy_versions`
   - `rollout_groups`
   - `rollout_episodes`
   - trainer job metadata
2. Write a proposal for the `Generator` and `TrainerAdapter` interfaces.
3. Decide what the first supported local serving topology should be.
4. Decide what the first supported external training framework should be.
5. Decide what fidelity level the first canonical trace implementation needs.

## Bottom Line

The right long-term architecture is likely:

- versioned asynchronous rollouts
- a separate generator service
- a separate trainer backend behind an adapter
- Arcane as the environment, trace, evaluation, and control plane
- lossless trace capture as the foundation for replay and post-training

That architecture gives Arcane a path from a benchmark/evaluation tool into a real asynchronous RL framework without collapsing everything into one monolithic runtime.
